In-memory Database

Nov 4th, 2011 | Comments

Definition

An in-memory database is a database that runs entirely in main memory, without touching a disk. Often they run as an embedded database: created when a process starts, running embedded within that process, and is destroyed when the process finishes.

(From Martin Fowler , InMemoryTestDatabase)

Contrast

Much faster than disk-optimized database, because simpler algorithms,fewer CPU instructions, fewer I/O reading activity. Lack support for durability portion of ACID, because it will lose all information when lose power or reset. But they can add durability via using some mechanisms, such as snapshot files,transaction logging,Non-volatile random, access memory, high availability.

When to use it ?

Using in-memory database in a enterprise application can dramatically reduce build time Not touching real database in unit test. it is safe for real database, keep database independent and clean. Don’t worry about the sequence of test caused different db result .

Structure of Function Test in Our Project

Nov 3rd, 2011 | Comments

Our project is a python web application based on Django framework, we use selenium as web driver to run our function test. Directory structure of function test is like this: –func_tests —-framework —-drivers —-utils —-pages —-indexpage —-projectpage … … —-resources —-testdata —-tests —-indexpagetests —-projectpagetests *******… …

Framework directory : In drivers directory: we wrap original web driver into costumed driver which offers more functions about choosing webdriver and selecting web elements . In utils directory: we offer basic operate functions for dealing with database, meanwhile, wrap basic web element to different specify elements such as button_web_element, drop_down_web_element.

pages directory: Create a single page object for each page you want to test, page object offers basic element locator 、navigation、response message and so on .

Resources directory: Put some local settings files for function test here, which are same as setting files in APP.

Testdata directory: Basic testdata for more than 1 function test , such as account info, initial date for database and so on.

Tests directory: All function tests are here, each page test is a single directory, also, end to end tests are here. In each test directory , there is page test file and corresponding test data.

Sequence of Find a Command in Shell

Nov 1st, 2011 | Comments

Do you know how the system understand which program to run when you type a command ?

Maybe you will say that system will look up the $PATH or check in the alias or somewhere else.

In the following link, it displays the detailed sequence.

How Linux or UNIX Understand which program to run – PART I

http://www.cyberciti.biz/tips/how-linux-or-unix-understand-which-program-to-run-part-i.html

Sequence(use date command):

  1. If there exists a shell FUNCTION date() execute it and stop.

  2. If there exists a shell builtin date command, execute it and stop

  3. If there exists an entry in hash table for date command execute it and stop.

  4. if there exists a bin file named date in PATH variable.

  5. If above all method fails then SHELL will return error “Command not found” and always exit with 127 status code

What’s hash table in bash shell ?

A HASH table is nothing but some sort of caching mechanism to speed up things. For each command, the full file name of the command is determined by searching the directories in $PATH variable and remembered by shell in HASH table. You can type hash command in your console to see the content of hash table, also, you could use “hash -r ” to remove all the memories.

Learned From Pair-programming

Aug 22nd, 2011 | Comments

刚刚进入公司时，没有预想中的培训，而直接进入了项目组，直接投入到Story的开发当中。作为新人，碰到了诸多的困难，不熟悉的开发环境（MAC和Intellij），全新的开发流程和风格（Agile），让人困惑的项目架构，各种前沿的技术知识，甚至还没有从学生的状态中转换过来。如何快速的融入到团队？如何快速的为项目作出贡献？最终如何快速的成长？也许Pair-Programming是个答案。

Pair-Programming

Pair Programming是一个编程模式(Programming pattern)。两个程序员(driver & observer)并排坐在一台电脑前，面对同一个显示器，使用同一个键盘，同一个鼠标一起工作。他们一起分析，一起设计，一起写测试例子，一起编码，一起单元测试，一起整合测试(Integration Test)，一起写文档等，基本上所有的开发环节都一齐肩并肩地，平等地，互补地进行开发工作。[1]

Pair-Programming所带来的好处包括更快的知识共享、较高的效费比（cost-effective）、减小人员流动带来的风险、高质量的代码、更强的问题解决能力、更多的信心和各方面的满足感。在Pair-Programming中所需要注意的问题也很多，比如及时的角色轮换，pair之前进行详细的任务分解，合理的分配pair，观点要保持统一等等。

当然，Pair-Programming也会被质疑到工作效率低，不同水平的人员之间难以配合，个人不同的开发习惯和最后的人空间等各种常见的问题。不管怎样，应该根据团队实际的情况出发，因为就像《人件》中所说的，人才是软件开发中最核心，也是最需要花时间去关注的事情。

What are you trying to learn?

Pair开始了，一开始是茫然的，看着Pair在键盘上跟飞似地，IDE跳来跳去，窗口切来切去，心里面那个羡慕啊。一开始都是在看Pair写代码，自己只能充当一个名义上的Observer，并且有时候不知道在看什么，常常会跑神。直到有一天知道，在pair的时候，常常问自己“What are you trying to learn?”，会让你更加清楚你想要什么，你在做什么，让pair更加的有效率。

What I have learned From Pair ?

从入职到现在的一个月里，和组里面的大多数人都Pair过，Pair不同的开发习惯和所擅长的知识领域让我受益匪浅。

总结收获如下：

1.良好的开发习惯

一开始常常出现如下的情况：

  • 当我终于可以拿着键盘和鼠标开始写代码的时候，就被blame了：要学着丢掉鼠标，使用快捷键来提高自己的开发效率，尤其是intellij里面各种用于重构的shortcut….

  • 当我攒了一大堆的代码准备提交的时候，就被打了：要及时的commit

  • 当我拿着Story卡就开始敲代码的时候，就被骂了：要先进行任务分解，清楚自己每一步都在干什么

  • 当我直奔着写方法和设计实现细节的时候，就被K了：要贯彻TDD的思想，小步快跑

2.写出高质量的代码

曾经感觉写的还可以的代码，敲出来就会被鄙视一顿，Pair总会告诉你有更好的方式来解决这个问题，哪些变量应该抽出去，哪部分应该写成接口，一步步的去除掉代码中的bad smell，逼着自己写出更好的代码。并且，会常常进行代码的重构，作为一个Pair的Observer会从更高的层面来对代码进行整理，好代码都是这么出来的。

3.知识的共享

Pair-Programming最大的好处，于是这一个月，跟索大师学BDD，跟超女学UI，跟超男学Chef，跟郑大学写高质量的代码，跟杨栋学习快速解决问题的方法，跟文博学习软件架构和重构。

从Pair身上可以学到太多自己不懂的知识和技术，更快速的扩展自己的知识边界。

4.对项目的理解和熟悉

在别的公司，新人进来往往有至少2个月的培训期，真正开始贡献代码往往都在3个月以后了。而通过和Pair一起写代码，能够非常快速的完成对项目代码的理解和熟悉，能够以最快的速度投入到开发中去。

5.程序员的素养

在Pair-Programming的过程中，一个人的知识技术水平、反映能力、解决问题的能力会非常清晰的暴露出来，变得公开和透明。

而像所说的程序员有四怕：

  • 怕自己看上去傻 

  • 怕被认为是没用的 

  • 怕自己变的不重要（过时)

  • 怕自己不够好。

Pair时会让人很有压力，会逼迫着你去学习、提升自己、证明自己。感谢和我一起pair过的同事，能够容忍我那么多低级的错误和傻笨的表现，我会继续努力，成为一个专业和成熟的TW的developer。

6.这是两个人的事

在开发的过程中，常常会碰见很多让人头痛的技术难题，当一个人面对时，往往会有很大的压力，常常会走极端，甚至会突然间的奔溃或抓狂。Pair让这些难题变成两个人的事情，既缓解了压力，并且增强了解决问题的能力。同甘苦、共患难，在开发过程中同样存在，能够增强对项目的责任心和团队整体的协作能力。

引用一个前辈的话：

我们Pair 做任何一件事情。 ——ThoughtWorks 西安Office

A Good Unit Test

Aug 14th, 2011 | Comments

Team上周的small talk的话题是一个好的单元测试应该是什么样子的，应该有什么样的准则，以下是我们所总结的其中一些rules。

一个测试包括：

  1. 预置条件（Given）

  2. 执行  (When)

  3. Assert （Then）

好的测试应该具备

  1. 简洁明了的断言

  2. 反应测试目的、容易理解的测试方法名

  3. 独立的测试，尽可能的减少测试之间的依赖性

  4. 逻辑简单，避免出现“if else”等等分支

  5. 排除测试中的不确定因素，如时间、多线程

  6. 减少对网络、文件、数据库等的依赖性

  7. 唯一的测试点

Story Development

Aug 14th, 2011 | Comments

Story development include the following steps:

Get story card

  1. you should analyze and understand the detail feature or function. If possible ,confirm the requirement with BA or QA.

  2. you should know the rationality of story, communicate with client.

  3. don’t spend too much time on the extrem or BT requirement .

Develop

  1. Decompose the story into several tasks which can be finished in limited time, but the granularity depends on you and your pairs.

  2. begin to write tests, using BDD and TDD to implement the function.

  3. Refractor your code and commit/push it.

Showcase

  1. Prepare the environment for the test according to AC.

  2. Test it at least 2 times by yourself.

  3. Show it to QA.
