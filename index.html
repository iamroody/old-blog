自动化脚本：添加用户

很多人看到这个题目时，肯定会心里暗说，这有什么困难的？然而在实际编写中，却遇到一些需要用户交互的过程，比如添加一个用户时，需要为其设置密码的情况。

以ubuntu为例，我们可以使用useradd或adduser来创建用户，关于这两个命令的区别，官方有提到：

useradd is a low level utility for adding users. On Debian,administrators should usually use adduser(8) instead.

下面是针对这两种命令的不同的自动化脚本来创建一个用户，并为其设置密码。

useradd

使用useradd添加用户时，可以通过-p选项来设置密码，然而此处的密码是经过加密的，因此需要借用openssl生成加密后的密码，也就是所看到的“$(echo yourpassword | openssl passwd -1 -stdin)”，该脚本意思是openssl从stdin读入密码string来生成MD5-based加密的密码。

再者，由于useradd是sh的命令，所以默认创建出的用户使用的也是“/bin/sh”, 可以通过“-s /bin/bash”来修改为bash。

adduser

使用adduser添加用户时，会直接提示需要输入密码，以及其他的诸如full name等信息，可以使用“–disabled-password”来跳过设置密码阶段，使用“–gecos ””来将用户其他信息设置为空。

用户创建成功之后，需要使用usermod来设置密码，此处依旧需要的是加密后的密码，所以还得openssl来显伸手。

约定

关于hashcode()有个通用的约定：
1. Whenever it is invoked on the same object more than once during an execution of a Java application, the hashCode method must consistently return the same integer, provided no information used in equals comparisons on the object is modified. This integer need not remain consistent from one execution of an application to another execution of the same application.
2. If two objects are equal according to the equals(Object) method, then calling the hashCode method on each of the two objects must produce the same integer result.
3. It is not required that if two objects are unequal according to the equals(java.lang.Object)) method, then calling the hashCode method on each of the two objects must produce distinct integer results. However, the programmer should be aware that producing distinct integer results for unequal objects may improve the performance of hashtables.
当然了，如果相同的实例能拥有唯一的hash code值，对hash table的性能提升有很大的帮助，固然很好，但是往往是蛮困难的。

PechaKucha

In thoughtworks university , every wednesday is PechaKucha night. At that night, presenter will give a presentation based on 20 slides, forward automatically in every 20 seconds. The topic can be anything that you know well or interested, such as drink, mind, movie star, food , tea, sports and so on.

In traditional presentation, presenter always talked a lot and miss the key point of subject. PechaKucha makes you using a simple and refined way to transfer new knowledge or concept , and it won’t take a long time ,just 400 seconds.

PechaKucha Night was devised in Tokyo in February 2003 as an event for young designers to meet, network, and show their work in public. It has turned into a massive celebration, with events happening in hundreds of cities around the world, inspiring creatives worldwide. Drawing its name from the Japanese term for the sound of “chit chat”, it rests on a presentation format that is based on a simple idea: 20 images x 20 seconds. It’s a format that makes presentations concise, and keeps things moving at a rapid pace.

In-memory Database

Definition

An in-memory database is a database that runs entirely in main memory, without touching a disk. Often they run as an embedded database: created when a process starts, running embedded within that process, and is destroyed when the process finishes.

(From Martin Fowler , InMemoryTestDatabase)

Contrast

Much faster than disk-optimized database, because simpler algorithms,fewer CPU instructions, fewer I/O reading activity. Lack support for durability portion of ACID, because it will lose all information when lose power or reset. But they can add durability via using some mechanisms, such as snapshot files,transaction logging,Non-volatile random, access memory, high availability.

When to use it ?

Using in-memory database in a enterprise application can dramatically reduce build time Not touching real database in unit test. it is safe for real database, keep database independent and clean. Don’t worry about the sequence of test caused different db result .

Structure of Function Test in Our Project

Our project is a python web application based on Django framework, we use selenium as web driver to run our function test. Directory structure of function test is like this: –func_tests —-framework —-drivers —-utils —-pages —-indexpage —-projectpage … … —-resources —-testdata —-tests —-indexpagetests —-projectpagetests *******… …

Framework directory : In drivers directory: we wrap original web driver into costumed driver which offers more functions about choosing webdriver and selecting web elements . In utils directory: we offer basic operate functions for dealing with database, meanwhile, wrap basic web element to different specify elements such as button_web_element, drop_down_web_element.

pages directory: Create a single page object for each page you want to test, page object offers basic element locator 、navigation、response message and so on .

Resources directory: Put some local settings files for function test here, which are same as setting files in APP.

Testdata directory: Basic testdata for more than 1 function test , such as account info, initial date for database and so on.

Tests directory: All function tests are here, each page test is a single directory, also, end to end tests are here. In each test directory , there is page test file and corresponding test data.
